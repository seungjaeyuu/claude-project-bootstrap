# claude-project-bootstrap 설계 원칙

> 본 플러그인은 실전 프로젝트에서 반복 시행착오로 결정화된 패턴을 **다음 프로젝트부터 기본값으로** 적용하기 위한 도구입니다. 여기서는 주요 설계 결정의 **근거** 를 기록합니다.

---

## 1. 꼭 지킬 것만, 강제력별 4단계

**원칙**: 규칙은 "하지 말 것" 위주로 쓰고, 프로젝트에 **특별히 해당되는 것만** 명시. 고성능 LLM (Opus 4.x+) 이 이미 판단 가능한 일반 모범 사례는 템플릿에서 제외. 각 규칙 앞에 이모지를 붙여 **강제력을 4단계로** 표시.

### 이 구조가 제공하는 것

- **CLAUDE.md 피로도 최소화**: 재서술하지 않아도 LLM 이 아는 규칙은 제외 → 문서 길이 절반 이하로 유지, 매 세션 읽히는 부담 감소
- **중요 규칙이 다수에 묻히지 않음**: 🚫 (절대 금지) vs 💡 (권장) 가 시각적으로 즉시 구분 → 위반 시 롤백 불가한 규칙이 일반 권장과 구별
- **프로젝트 고유 학습 포인트 집중**: 사고·시행착오 경험을 압축해서 가장 먼저 전달 — 신규 참여자·AI 가 위험 영역을 빠르게 학습

### 4단계

| 이모지 | 단계 | 의미 |
|---|---|---|
| 🚫 | Guardrail (절대 금지) | 롤백 불가한 피해 또는 재발된 사고의 근거 — override 불가 |
| 📐 | Schema (계약) | 스키마·용어·명명 규약 — 일관성 확보 목적 |
| 📎 | 참조 | 경로·환경·도구 메타 정보 — 지침 아님 |
| 💡 | 권장 | 모범 사례 — 근거 제시 시 override 가능 |

**적용**: `templates/CLAUDE.md.tmpl` 상단 범례 표.

**배경**: 일반 모범 사례를 CLAUDE.md 에 나열하면 고성능 LLM 에게는 노이즈가 되고, 정작 프로젝트 고유의 위험 영역이 묻히는 문제가 실증되어 "네거티브 우선" 구조로 정착했습니다.

### 컨텍스트 윈도우는 유한 자원 (v0.3.0 부터)

임계치의 근거는 "가독성"만이 아니다. Claude Code 의 컨텍스트 윈도우(~200K 토큰)는 유한 자원이고, 매 세션 자동 로드되는 항목(CLAUDE.md + 활성 플러그인 + MCP 서버)이 이 예산을 소비한다. v0.3.0 기준 플러그인+MCP 만으로 ~19%(~38,800 토큰)가 소비되는 것으로 측정되었다. CLAUDE.md 가 비대하면 이 예산이 추가로 줄어들어 실제 작업에 쓸 컨텍스트가 부족해진다.

이 관점에서 v0.3.0 은 세 가지 메커니즘으로 컨텍스트 예산을 관리한다:
- **줄 수 임계치**: CLAUDE.md 120줄 / RULES 250줄 (v0.2.0 부터)
- **enabledPlugins**: `.claude/settings.json` 에서 불필요한 플러그인을 `false` 로 설정하면 해당 플러그인이 0 토큰을 소비. `/audit --context` 가 최적화 자동 제안
- **.claudeignore**: Claude Code 의 파일 탐색 대상에서 바이너리·빌드 산출물을 제외하여 불필요한 파일 읽기 차단

### 메커니즘으로 임계치 강제 (v0.2.0 부터)

CLAUDE.md 비대화는 *문서 규칙*만으로는 막히지 않는다. 시간이 지나면서 항목이 추가되고 어느 시점부터 LLM 의 매 세션 부담이 임계치를 넘는다. 이 문제를 한 가지 메커니즘으로만 막기로 한다 — 줄 수 측정 hook.

#### 이 구조가 제공하는 것

- **상한이 명시적**: 120줄(본체) / 250줄(sub) — 추상적 "간결하게" 가 아니라 측정 가능한 숫자
- **자동 알림 + 수동 결정**: pre-commit 이 경고만 주고 차단 안 함 — 임계치 초과의 합리적 사유가 있을 때 사용자가 결정. 다만 "잊혀짐" 은 차단됨
- **자기 위반 회피**: 임계치 자체를 LLM 본체에 또 적지 않음 — 메커니즘이 이미 강제하므로 본체에 추가하면 SSOT 자기 위반
- **경계 결정의 출발점**: 본체 vs 분리의 의사결정이 줄 수 측정 결과 → §4 "발견 시점 설계" 의 트리거 표 추가/제거 결정으로 자연스럽게 이어짐

#### 임계치

| 대상 | 임계치 | 비고 |
|---|---|---|
| `CLAUDE.md` 본체 | 120줄 | `scripts/check_doc_size.py` 경고 |
| `apps/*/CLAUDE.md` | 250줄 | 동일 스크립트 |
| `docs/rules/RULES_*.md` | 250줄 (분리 시점 권장 200줄 이하) | 동일 스크립트 |

**배경**: v0.1.x 의 Full tier 가 298줄까지 비대해진 사례 — 매 세션마다 그만큼의 토큰을 LLM 이 attention 처리하게 됨. 이 회귀가 본 플러그인의 "토큰 절약을 통한 개발 정확성" 목적의 자기 위반이라는 인식에서 v0.2.0 슬림화 정책이 정착했습니다.

## 2. 테스트 항목 3가지 표준 서식

**원칙**: 프로젝트의 테스트 항목 목록 (`{APP}_MASTER_TEST_BASELINE.md`) 을 **3개 표준 서식 (Template A/B/C)** 중 하나로 관리. 의미가 같은 것을 다른 이름으로 부르지 않도록 표 구조를 통일.

### 이 표준화가 제공하는 것

- **사람·스크립트 양쪽 가독성**: 동일 컬럼 구조 → `baseline_status.py` 가 파싱 실패 없이 집계하고, 사람도 표 헤더를 매번 학습할 필요 없음
- **테스트 항목의 역할 구분 명확**: Template A (실행 완료) / B (미실행·보강 필요) / C (버그 추적, 집계 제외) 로 각 항목의 의도가 표에 드러남
- **협업 부담 감소**: 여러 사람·AI 가 같은 프로젝트를 만져도 표 구조 토론 없이 즉시 편집 가능
- **신규 항목 추가 비용 최소**: 어느 Template 을 쓸지만 결정하면 나머지는 기계적

### 3 Template

| Template | 대상 |
|---|---|
| A | 이미 실행 완료된 테스트 ID |
| B | 미실행·소스 보강 필요 |
| C | 버그 추적용 (집계 제외) |

**기존 데이터 취급**: 일괄 변환하지 않음 — 자연어 셀을 재해석할 때 정보 손실 위험. 편집 기회가 생길 때마다 점진 전환.

**적용**: `templates/BASELINE.md.tmpl` 이 3 Template 구조로 작성되어 있음.

**배경**: 실제 프로젝트의 베이스라인 파일에서 13가지 서로 다른 헤더 변형이 발견되어 (예: `항목` / `미실행 필수 항목` / `기준 항목`) 스크립트 파싱이 불가능했던 사례를 바탕으로, 3개 표준으로 정착했습니다.

## 3. 테스트 결과와 문서 설명 분리

**원칙**: 베이스라인 (테스트 항목 목록) 의 **판정** — 통과/실패/보류 (PASS/FAIL/BLOCK) — 를 구조화된 `status.json` 파일에 단일 원천 (SSOT) 으로 보관. baseline markdown 은 과거 맥락 요약 역할로 분리.

### 이 분리가 제공하는 것

- **항목별 명확한 상태 관리**: 판정이 JSON 구조로 존재하므로 `baseline_status.py` 가 즉시 집계 — 누가·언제·어떤 ID 가 어떤 판정을 받았는지 파싱 로직 없이 확인
- **여러 세션·테스터 결과의 통일성**: 동일 포맷 강제 → 사람·AI 해석 편차 제거, 교차 검증 용이
- **수정 개발 항목의 자동 도출**: `baseline_status.py --failed` / `--blocked` 쿼리로 작업 큐가 자연스럽게 드러남 → 릴리스 전 해결해야 할 항목이 즉시 목록화
- **영역별 품질 추이 측정**: `--by-prefix` 로 기능 영역별 (AUTH/NOT/PRV 등) PASS 비율 집계 → 릴리스 품질 지표, 회고 근거로 활용

### 적용

| 데이터 | 위치 |
|---|---|
| 판정 (PASS/FAIL/BLOCK) | `docs/<app>/testing_harness/run/*_status.json` |
| 요약 (narrative) | baseline markdown "최신 요약" 컬럼 (판정 없이) |
| 후속 과제 | 프로젝트의 백로그 시스템 (GitHub Issues 등) |
| 참조 | baseline markdown "이전 프롬프트 매핑" 링크 |

**도구**: `scripts/baseline_status.py` 가 `status.json` 만 집계. baseline markdown 은 사람이 읽는 정보 보관·참조용.

**배경**: 한 셀에 판정·메모·후속과제·참조 4가지를 같이 기록하면 자동화가 불가능함이 실증되어, 역할별로 파일·컬럼을 분리한 구조로 정착했습니다.

## 4. 규칙은 한 곳에만, 필요할 때 발견

**원칙**: 같은 규칙을 여러 파일에 복붙하지 않고, 하나의 출처만 유지. 대신 **"어느 상황에서 Claude 가 그 규칙을 Read 하게 되는가"** 를 설계.

### 이 구조가 제공하는 것

- **동기화 실패 제거**: 하나의 출처만 편집하면 됨 → 복붙된 규칙 간 내용 불일치 (divergence) 가 원천 차단
- **세션 인지 부담 최소화**: 매 세션마다 모든 규칙을 한꺼번에 읽게 하지 않음. 각 규칙이 **필요한 시점에** 자동 노출되도록 배치
- **자연스러운 점진 학습**: 전사 규칙 (Git 안전 등) 은 매 세션, 영역별 규칙 (앱별) 은 해당 디렉토리 작업 시, 상세 참조는 요청 시 로드
- **유지보수 경로 일원화**: 규칙 변경 시 "어디 어디를 동시에 수정해야 하는가" 가 없음 — 단일 파일 수정

### 발견 단계 (강제력 순)

| 단계 | 메커니즘 | 등장 시점 |
|---|---|---|
| T1 | Hook (자동 실행 스크립트) | 가장 확실 — Git pre-commit, Claude Code PostToolUse 등이 기계적으로 강제 |
| T2 | 자동 로드 (루트 `CLAUDE.md`) | 매 세션 시작 시 인지 |
| T3 | 조건부 로드 (앱별 `CLAUDE.md`) | 해당 영역 작업 시 |
| T4 | On-demand (`TESTING_FRAMEWORK.md` 등) | Claude 가 필요에 따라 Read |

**적용 (이 플러그인의 구조)**: `commands/` (사용자가 명시적 호출), `templates/` (스캐폴드 시 1회 복사), `scripts/` (target 프로젝트로 복사 후 hook 에서 자동 호출) 로 발견 시점을 분리 배치.

**배경**: 동일 규칙을 여러 파일에 복붙했을 때, 한 파일만 업데이트되면서 파일 간 내용이 달라져 혼란이 증폭되는 사례가 반복되어, 단일 출처 + 발견 시점 설계 구조로 정착했습니다.

### 분리만으로는 충분치 않다 — 발견 트리거 표 (v0.2.0 부터)

영역별 상세 규칙을 별도 파일로 옮기는 것 자체로는 토큰을 절약할 수 없다. LLM 이 안전을 위해 모든 RULES 를 매번 read 하면 분리 의미가 사라지고, 안 read 하면 잘못된 코드가 나와 수정개발이 더 발생한다. **본체의 트리거 표가 "어느 작업이면 어느 RULES 를 read 하라" 를 명시적으로 매핑**하는 게 핵심이다.

#### 이 구조가 제공하는 것

- **세션 시작 시 본체만 attention**: 매 세션 자동 로드되는 본체는 횡단 가드레일 + 트리거 표만 — 세션 단위 토큰 절감
- **작업 시점에 정확한 1개 read**: LLM 이 작업 종류를 판단해 매칭되는 RULES 1개만 read. 영역별 깊이는 가져가되 세션 토큰은 감소
- **fallback 으로 누락 차단**: 트리거 표에 해당 안 되는 작업은 `design-principles.md` 로 회귀 — RULES 미read 로 인한 사고 위험 방지
- **확장 비용 작음**: 새 영역 추가는 트리거 표 1행 + RULES 파일 1개. 본체 비대화 없음

#### 영역 매핑

| 영역 | 분리 파일 | 발견 트리거 |
|---|---|---|
| E2E 테스트 / Codex orchestrator | `RULES_E2E.md` | E2E·orchestrator 작업 |
| 백엔드 데이터 정합성 | `RULES_DATA_INTEGRITY.md` | Firestore / 백엔드 호출 코드 |
| Accessibility identifier | `RULES_ACCESSIBILITY.md` | UI 컴포넌트 편집 |
| 도메인 용어 / 스키마명 병기 | `RULES_TERMINOLOGY.md` | 도메인 용어 등장 코멘트·UI 텍스트 |
| Dictionary 중복 키 | `RULES_DICT_DUPLICATES.md` | 매핑·번역·설정 dict 편집 |
| 대규모 변경 / 리팩토링 | `RULES_REFACTORING.md` | 100줄 이상 변경 / 리팩토링 |
| (분류 모호 fallback) | `docs/design-principles.md` | 위 트리거 미적용 작업 |

**배경**: v0.2.0 설계 단계에서 "본체 슬림화만 하면 충분한가?" 를 점검하다가 *분리만으로는 토큰 절약이 보장되지 않음*을 발견했습니다. 트리거 표를 본체에 명시하는 단계가 추가되어 분리·발견·재회귀의 3박자가 맞게 정착했습니다.

## 5. 설정 한 줄로 테스트 자동화 시작

**원칙**: E2E 테스트 하네스 (AI 워커가 실행하는 테스트 자동화) 의 **"도입 시점"** 을 모호하지 않게 만들기 위해, `scripts/baseline.yml` 의 `apps.<name>` 항목 활성화를 **단일 스위치**로 삼음.

### 이 구조가 제공하는 것

- **도입 시점의 명확화**: "하네스 도입" 이 사람마다 다른 해석을 갖지 않음 — entry 활성 여부 = 도입 여부
- **하나의 스위치로 일괄 인식**: `baseline.yml` 한 항목만 활성화하면 스크립트·hook·도구가 자동으로 해당 앱을 대상에 포함
- **신규 앱 추가 비용 최소**: 코드 수정 없이 설정 파일에 entry 한 줄 추가만으로 완료
- **백로그 정리 용이**: "언제 실행할지" 모호한 항목이 사라짐 — 활성화된 것만 실행 대상

### 적용

`templates/baseline.yml.tmpl` 은 주석 처리된 (commented-out) 엔트리 + 활성화 설명으로 구성. 실제 프로젝트는 필요한 앱만 주석 해제해 사용.

```yaml
default_app: ios

apps:
  ios:
    baseline: IOS_MASTER_TEST_BASELINE.md
    status_dir: docs/ios/testing_harness/run
```

## 6. 정해진 시점에 일괄 검토

**원칙**: 베이스라인 갱신을 매 커밋·매 기능 수정마다 판단하지 않음. 정해진 시점 (트리거) 에 Git diff 를 자동 분석해 변경 사항을 제안받고, 사용자 승인 후 일괄 반영.

### 이 구조가 제공하는 것

- **릴리스 사이클과 자연 동기화**: E2E 세션 시작·릴리스 준비·주간 점검 같은 자연스러운 시점에만 검토 → 개발 흐름을 끊지 않음
- **매번 판단 부담 제거**: 매 프롬프트마다 "이 변경이 baseline 갱신 대상인가?" 를 판단할 필요 없음
- **정확도 향상**: 사람·AI 가 한 번에 여러 변경을 모아 검토 → 개별 즉시 판단보다 전체 맥락에서 정확한 분류 가능
- **강제력의 단계적 구성**: 놓침은 경고로 알려주되 차단은 하지 않아 **유연성과 안전성 균형**

### 강제력 단계

| 단계 | 메커니즘 | 차단 여부 |
|---|---|---|
| L1 | Git pre-commit hook 경고 | 차단 안 함 (놓침 방지 목적) |
| L3 | `/baseline-review` 명시 호출 | — (사용자 트리거) |
| L4 | 문서 규약 (`TESTING_FRAMEWORK.md §20.7`) | — (참조) |

### 트리거 예시

- E2E 테스트 세션 시작
- 릴리스·베타 준비
- 정기 점검 (주 1회 권장)

**도구**: `/claude-project-bootstrap:baseline-review` 슬래시 커맨드 → `scripts/baseline_update_suggest.py` 호출 → Git diff 분석 → 신규/수정/폐기 항목 제안.

**배경**: "매 기능 수정마다 baseline 갱신" 을 시도했을 때 매 프롬프트에서 판단을 요구하게 되어 거짓양성·음성이 다수 발생했습니다. 반대로 개별 커밋마다 일일이 갱신하면 비효율이 컸습니다. 릴리스 사이클과 맞춰 일괄 검토하는 구조로 정착했습니다.
