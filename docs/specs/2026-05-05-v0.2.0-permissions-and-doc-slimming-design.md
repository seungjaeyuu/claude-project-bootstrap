## 1. 배경과 목적

### 1.1 이번 업데이트의 궁극적 목적

본 플러그인은 **개발 정확성을 통한 토큰 절약**을 궁극적 목적으로 한다. 부정확한 개발은 수정개발·추가개발을 유발하고, 그 자체가 토큰 비용이다. v0.1.x 에서 `templates/CLAUDE.md.tmpl` 이 298줄까지 비대해진 것은 본 목적의 자기 위반이며, v0.2.0 은 그 회귀를 시작점으로 삼는다.

### 1.2 사용자 요청

1. **Bash 명령 권한 수준을 옵션화** — "완전 자율" 부터 "미설정(기본 동작)" 까지 init 시점에 선택. 사용자가 자율 주행 강도와 보안 강도를 프로젝트 시작 시 의식적으로 결정하게 한다.
2. **CLAUDE.md 최적화 규칙(임계치 등) 및 문서 작성 규칙(계층화, SSOT) 도입** — 본체 비대화를 메커니즘으로 차단.
3. **(브레인스토밍 중 추가) Firebase 격리 안전장치** — "기본 프로젝트가 다른 프로젝트로 상속되어 잘못된 곳에 deploy" 사고 재발 방지. CLI 가 4단계 우선순위(`--project` > 환경변수 > `.firebaserc` > 글로벌 캐시) 로 활성 프로젝트를 결정한다는 점을 활용.

### 1.3 비-목적 (이번 범위 외)

- Supabase / AWS Amplify / 기타 BaaS 의 격리 안전장치 — Q2 Yes 에서 Firebase 외 백엔드는 안내 섹션만 유지, 격리 메커니즘은 미구현.
- Bash permission 단계의 사용자 정의 미세 조정 UI — 사용자가 후에 `.claude/settings.json` 직접 편집해 조정.
- 이미 분리된 `ACCESSIBILITY_IDENTIFIERS.md` (프로젝트별 카탈로그) 는 변경 없음. 이번 분리는 **규칙 본문**(`RULES_ACCESSIBILITY.md`) 에 한정.

### 1.4 자기 위반 회피 원칙 (이 문서 자체에 적용)

본 디자인은 다음 4개 자기 위반을 회피한다 (브레인스토밍 단계의 비판적 재점검 결과):

1. **본체 inline 메타-규칙 추가 금지** — `design-principles.md` §1·§4 와 중복되는 내용을 `CLAUDE.md` 본체에 또 적지 않는다. SSOT 자기 위반.
2. **Hook 이 강제하는 것을 LLM 에게 말로 설명 금지** — 임계치는 `check_doc_size.py` 가 강제하므로 LLM 본체 인지 불요.
3. **메커니즘이 자동 차단하는 것을 LLM 에게 말로 설명 금지** — Firebase predeploy hook 이 자동 검증하므로 본체 안내 최소화.
4. **분리만 하지 말고 발견 시점 명시** — RULES 분리 후 *언제 read 하는가* 를 본체 트리거 표로 매핑. 분리 자체는 토큰 절약을 보장하지 않는다.

## 3. Bash 권한 단계 (Q0)

### 3.1 단계 정의

각 단계는 init-project 시점에 다음 표 그대로 사용자에게 노출한다. 한 줄 요약 + 사용자 체감 예시 1줄 구조.

| 단계 | 한 줄 요약 | 사용자 체감 예시 |
|---|---|---|
| **YOLO** | 거의 모든 Bash 자동 실행. 파괴 명령(`rm -rf`, `git reset --hard`, `git push --force` to main, DB drop 류)만 deny | "프로토타입 빨리 돌리고 싶고, 위험 명령은 안 쓸게" |
| **Standard** *(권장)* | 읽기·일반 빌드 명령은 자동, 파일 삭제·git 변경·deploy·DB 마이그레이션은 ask | 보통 작업은 끊김 없이, 위험 명령에서만 한 번 묻기 |
| **Strict** | 거의 모든 Bash 가 ask. 읽기 전용(`ls`, `cat`, `git status`, `git diff` 등)만 자동 | "Claude 가 뭘 할 때마다 일단 보고 싶음" / 보안 민감 |
| **None** | `permissions` 키 자체를 만들지 않음 (Claude Code 기본 동작) | "기본값으로 충분, 직접 안 건드림" |

### 3.2 SSOT 위치

- 플러그인: `templates/permissions/{yolo,standard,strict}.json` 3개 (None 은 파일 없음, init-project 가 `permissions` 키를 settings.json 에 머지하지 않는 분기로 처리)
- target 프로젝트: `.claude/settings.json` 의 `permissions` 키

### 3.3 단계별 JSON 조각 설계 원칙

각 JSON 조각은 `permissions: { allow, deny, ask }` 3개 키만 포함. 확장 키는 두지 않는다.

**yolo.json (요지)**
```json
{
  "permissions": {
    "allow": ["Bash(*)"],
    "deny": [
      "Bash(rm -rf /:*)",
      "Bash(rm -rf ~:*)",
      "Bash(git reset --hard:*)",
      "Bash(git push --force:* main)",
      "Bash(git push --force:* master)",
      "Bash(git push -f:* main)",
      "Bash(git push -f:* master)",
      "Bash(npx prisma migrate reset:*)",
      "Bash(dropdb:*)",
      "Bash(psql:*-c \"DROP*)"
    ]
  }
}
```
- `firebase deploy` 는 deny 에 두지 않는다 (롤백 가능, 일반 배포 작업). 잘못된 프로젝트로 deploy 사고는 §4.2 의 `predeploy` hook (`scripts/check_firebase_project.py`) 이 차단한다 — deny 와 hook 의 이중화는 사용자가 명시적으로 deploy 를 지시한 케이스까지 막아 over-restriction 이 된다 (v0.2.0 → v0.2.1 amend, 2026-05-08).

**standard.json (요지)**
```json
{
  "permissions": {
    "allow": [
      "Bash(ls:*)", "Bash(cat:*)", "Bash(grep:*)", "Bash(find:*)",
      "Bash(git status:*)", "Bash(git diff:*)", "Bash(git log:*)", "Bash(git show:*)",
      "Bash(npm run build:*)", "Bash(npm run lint:*)", "Bash(npm test:*)",
      "Bash(python3:*)", "Bash(node:*)",
      "Bash(mkdir:*)", "Bash(touch:*)",
      "Bash(./gradlew:*)", "Bash(xcodebuild:*-list)"
    ],
    "ask": [
      "Bash(rm:*)", "Bash(mv:*)", "Bash(cp:*-rf*)",
      "Bash(git commit:*)", "Bash(git push:*)", "Bash(git reset:*)",
      "Bash(git checkout:*)", "Bash(git merge:*)", "Bash(git rebase:*)",
      "Bash(firebase:*)", "Bash(npx supabase:*)",
      "Bash(npm install:*)", "Bash(npm uninstall:*)",
      "Bash(npx prisma migrate:*)"
    ],
    "deny": [
      "Bash(rm -rf /:*)", "Bash(rm -rf ~:*)",
      "Bash(git reset --hard:*)",
      "Bash(git clean -fd:*)", "Bash(git clean -fdx:*)",
      "Bash(git push --force:* main)", "Bash(git push --force:* master)",
      "Bash(git push -f:* main)", "Bash(git push -f:* master)"
    ]
  }
}
```
- 일반 빌드/lint/test 는 자동, 파일·git 변경·deploy·패키지 변경은 ask, 절대 파괴는 deny.
- `git reset --hard` 는 ask 가 아닌 **deny** — 사용자 CLAUDE.md 공통 규칙 (`~/.claude/CLAUDE.md`) 에서 절대 금지로 지정된 사항이라 ask 도 허용 안 함.
- `git clean -fd` 는 untracked 파일 삭제 — 복구 불가하므로 deny.
- ask 와 deny 의 경계 원칙: **롤백 가능한 위험은 ask, 롤백 불가한 위험은 deny**. 동일 원칙을 yolo.json / strict.json 에도 적용.

**strict.json (요지)**
```json
{
  "permissions": {
    "allow": [
      "Bash(ls:*)", "Bash(cat:*)", "Bash(grep:*)", "Bash(find:*)",
      "Bash(git status:*)", "Bash(git diff:*)", "Bash(git log:*)", "Bash(git show:*)",
      "Bash(pwd:*)", "Bash(echo:*)"
    ],
    "deny": [
      "Bash(rm -rf /:*)", "Bash(rm -rf ~:*)",
      "Bash(git push --force:* main)", "Bash(git push --force:* master)"
    ]
  }
}
```
- 읽기 전용만 allow. 그 외 전부 ask (Claude Code 기본 동작이 unknown 명령에 ask).

### 3.4 init-project Q0 흐름

```
[필수 1-4 입력 후]

Q0. Bash 권한 단계
   다음 표를 보고 1개 선택:
   1) YOLO     (거의 모두 자동, 파괴 명령만 deny)
   2) Standard (권장: 안전 명령 자동, 위험 명령 ask)
   3) Strict   (읽기 전용만 자동, 그 외 ask)
   4) None     (Claude Code 기본 동작)

   상세 표:
   [위 3.1 표 그대로 출력]
```

선택값에 따라:
- 1-3: `templates/permissions/{yolo,standard,strict}.json` 의 `permissions` 키를 `.claude/settings.json` 에 머지 (기존 `hooks` 키 보존)
- 4: `permissions` 키 미추가

### 3.5 hooks 와의 머지 규칙

- Q4 Yes (Accessibility ID 검증) 시 `templates/settings.json.tmpl` 의 `hooks` 키가 `.claude/settings.json` 에 들어간다.
- Q0 의 `permissions` 키는 별개 키이므로 충돌 없음. 동일 파일에 공존.
- Q4 No 이고 Q0 != None 이면 `.claude/settings.json` 에 `permissions` 키만 있는 파일 생성.
- Q4 No 이고 Q0 == None 이면 `.claude/settings.json` 자체를 만들지 않는다.

## 4. Firebase 격리 안전장치

### 4.1 Q2 흐름 변경

```
Q2. BaaS 사용? (Yes/No)
  Yes →
    Q2a. 어떤 백엔드?
       1) Firebase
       2) Supabase
       3) AWS Amplify
       4) 기타
    Q2b. (Q2a == 1 일 때만) Firebase project ID? (예: aidea-prod)
```

Q2a == 2/3/4 인 경우는 v0.1.x 와 동일 동작 (보안 섹션 + `.env.example`). 격리 메커니즘은 Firebase 한정.

### 4.2 생성 파일 (Q2a == Firebase 시)

#### 4.2.1 `.firebaserc` (자동 생성)

```json
{
  "projects": {
    "default": "<Q2b 입력값>"
  }
}
```
- 단일 alias `default` 만 작성. `production`/`staging` 같은 추가 alias 는 사용자 후속 설정 (LLM 자력 판단).
- 이게 활성 프로젝트 결정의 SSOT (CLI 우선순위 #3, 글로벌 캐시 #4 보다 우선).

#### 4.2.2 `firebase.json` (생성 또는 머지)

기존 파일이 있으면 `predeploy` 키만 머지. 없으면 minimal 생성.

```json
{
  "functions": {
    "predeploy": [
      "python3 scripts/check_firebase_project.py"
    ]
  },
  "hosting": {
    "predeploy": [
      "python3 scripts/check_firebase_project.py"
    ]
  },
  "firestore": {
    "predeploy": [
      "python3 scripts/check_firebase_project.py"
    ]
  },
  "storage": {
    "predeploy": [
      "python3 scripts/check_firebase_project.py"
    ]
  }
}
```

- 4개 영역(functions/hosting/firestore/storage) 각각의 `predeploy` 에 검증 스크립트 등록.
- 사용자가 사용 안 하는 영역은 entry 자체 삭제 가능 (init-project 가 모두 등록, 사용자 후속 정리).
- `firebase deploy --only functions` 처럼 한 영역만 deploy 해도 그 영역의 predeploy 가 실행되어 검증 누락 없음.

#### 4.2.3 `scripts/check_firebase_project.py`

검증 로직:
1. `.firebaserc` 읽기 → `projects.default` 추출 (= 기대 ID)
2. 활성 프로젝트 확인:
   - 우선 `firebase use` 명령 출력 파싱 (stdout 의 active 프로젝트)
   - 명령 미설치 시: `~/.config/configstore/firebase-tools.json` 의 `activeProjects` 에서 현재 디렉토리 매핑 조회
3. 환경변수 `GCLOUD_PROJECT` / `GOOGLE_CLOUD_PROJECT` 설정 시: 그것이 우선
4. 일치하지 않으면 stderr 에 명확한 에러 메시지 + exit 1
5. 일치하면 stdout 한 줄 ("✅ Firebase project = <ID>") + exit 0

에러 메시지 예시:
```
🚫 Firebase 프로젝트 mismatch
   .firebaserc default: aidea-prod
   현재 활성: forkfork-dev
   해결:
     firebase use aidea-prod
   또는 deploy 명령에 --project aidea-prod 명시
```

## 5. CLAUDE.md 슬림화 + 영역별 분리

### 5.1 임계치

- **본체 (`CLAUDE.md`)**: 120줄 — `check_doc_size.py` 가 경고 (차단 안 함)
- **sub-CLAUDE.md (`apps/*/CLAUDE.md`)**: 250줄

### 5.2 Full tier 재구성 (목표 100-115줄)

#### 5.2.1 본체에 남기는 섹션 (총 ~70-80줄 + 조건부 옵션)

| 섹션 | 라인 수 | 내용 |

## 9. 호환성 + 기능별 재설정·도입 커맨드

### 9.1 핵심 정책

1. **기존 v0.1.x 프로젝트는 그대로 작동** — 플러그인 업데이트만으로 자동 변경되지 않음. 사용자가 v0.1.x 로 init 한 CLAUDE.md / settings.json / scripts 는 v0.2.0 출시 후에도 무수정 사용 가능.
2. **새 기능 도입은 기능별 슬래시 커맨드로 처리** — 단일 마이그레이션 커맨드 대신 영역별 4개 커맨드 제공. 사용자는 필요한 영역만 선택 적용.
3. **이 커맨드들은 마이그레이션 전용이 아님** — *재설정·재도입* 양쪽 용도. 신규 프로젝트에서 init 후 후속 변경(예: Bash permission 단계 변경)에도 동일 커맨드 사용.

### 9.2 4개 기능별 커맨드 사양

| 커맨드 | 용도 | 백업 대상 |
|---|---|---|
| `/claude-project-bootstrap:bash-permission` | YOLO/Standard/Strict/None 단계 도입·변경 | `.claude/settings.json` |
| `/claude-project-bootstrap:firebase-isolation` | Firebase 격리 도입 (project ID 입력 → .firebaserc + firebase.json predeploy + check 스크립트 + CLAUDE.md 가드) | `.firebaserc`, `firebase.json`, `CLAUDE.md` |
| `/claude-project-bootstrap:slim-claude-md` | CLAUDE.md 슬림화 + RULES 분리 (진단 → 사용자 확인 → 분리) | `CLAUDE.md`, `INDEX.md` |
| `/claude-project-bootstrap:doc-size-hook` | check_doc_size.py 도입 + pre-commit 통합 | `scripts/`, `.git/hooks/pre-commit` |

### 9.3 4중 안전장치 (모든 커맨드 공통)

각 커맨드는 다음 4개 안전장치를 내장한다:

1. **자동 백업**: 변경 대상 파일을 `_backup_v0.1/<원본 경로>` 로 *먼저 복사*. `_backup_v0.1/` 는 `.gitignore` 자동 추가 (커밋 오염 방지). 사용자는 언제든 복원 가능.
2. **Diff preview**: 변경 적용 전 사용자에게 "이렇게 바뀐다" diff 출력 → yes/no 확인. 거부 시 그 영역 skip.
3. **영역별 독립 적용**: 각 커맨드가 독립이므로 사용자가 A·B·D 만 도입하고 C 는 skip 가능. 한 커맨드 안에서도 sub-옵션 (예: firebase-isolation 의 "predeploy hook 도 자동 등록할까?") 은 독립 yes/no.
4. **본체 자동 변경 회피 — 진단·확인 기반만**: `slim-claude-md` 의 본체 분리는 *자동 분리하지 않음*. `scripts/migrate_diagnose.py` 가 현재 CLAUDE.md 를 분석해 분리 후보 진단 리포트 출력 → 사용자가 후보별 yes/no → yes 한 후보만 분리. 본체에서 자동 삭제도 매 후보별 confirm.

진단 리포트 예시 (`slim-claude-md` 가 호출):
```
📊 분리 후보 진단

CLAUDE.md 현재 라인 수: 287줄 (임계치 120줄 초과)

분리 후보 (큰 섹션 순):
  [1] §8 E2E 테스트 (현재 80줄)        → RULES_E2E.md 로 분리 권장 (yes/no?)
  [2] §6 데이터 정합성 (현재 22줄)      → RULES_DATA_INTEGRITY.md 로 분리 권장 (yes/no?)
  [3] §2 Accessibility (현재 18줄)     → RULES_ACCESSIBILITY.md 로 분리 권장 (yes/no?)
  [4] §4 용어 규칙 (현재 16줄)         → RULES_TERMINOLOGY.md 로 분리 권장 (yes/no?)
  [5] §2 dict 중복 키 (현재 8줄)       → 작아서 분리 불요, 본체 유지

각 후보를 분리하시겠습니까?
```

### 9.4 Breaking Change 없음

- 슬래시 커맨드 이름·인터페이스 v0.1.x 와 동일 (`init-project`, `baseline-review` 변경 없음)
- 기존 `templates/CLAUDE.md.tmpl` 슬림화는 *새 프로젝트* 에만 자동 적용
- 기존 v0.1.x 로 init 한 프로젝트는 영향 없음 (사용자가 명시적으로 4개 커맨드 중 하나를 호출할 때만 변경)
